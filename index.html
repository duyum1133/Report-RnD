RabbitMQ Basic

I. What is RabbitMQ? How does RabbitMQ work

Event Queue is an important design pattern that helps analyze a monolithic application. It postpones sending events to a point in the execution flow where they can be safely processed.

1. What is RabbitMQ?

RabbitMQ is an open source message broker written in Erlang.

RabbitMQ was originally developed in collaboration between LShift, LTD and Cohesive FT, now owned by Pivotal Software Inc and released in 2007 under the Mozilla Public License.

RabbitMQ supports the publish/subscribe model, looping model and request/response model. It provides features such as automatic recovery, routing and message filtering. RabbitMQ supports many programming languages such as PHP, Java, .NET, Ruby, Python, etc.

RabbitMQ was one of the first message brokers to implement the AMQP (Advanced Message Queuing Protocol) protocol.

AMQP is an open (open protocol) standardized communication protocol for reliably and efficiently transmitting messages between applications and message brokers. AMQP provides important features such as bandwidth limits, load limits, and automatic message forwarding to another system if the original system cannot handle it.

2. How does RabbitMQ work?

The message processing flow in RabbitMQ works as follows:

  1. The producer sends a message to an exchange. The exchange type must be specified when it is created.
  1. The exchange receives the message and is responsible for routing the message. The exchange will consider the message properties and exchange type, such as routing key.
  1. Bindings must be established between the exchange and queue in advance. The exchange routes the message to the queue depending on the message properties. In the above example, there are 2 bindings from the exchange to 2 different queues.
  1. Messages remain in the queue until they are processed by consumers.
  1. The consumer processes the message.

3. Main components of RabbitMQ

In RabbitMQ's architecture, the main components include messages, producers, exchanges, bindings, queues, and consumers.

3.1. Message

Message is the basic unit for exchanging information in RabbitMQ. It includes a body (also called payload) and optional other properties such as header, routing key, message ID, etc.

3.1.1. Message Body

Message body contains the actual data to be sent, which can be in various formats such as JSON, XML, binary or any structure. The message body content can represent commands, events, notifications or any information to be exchanged.

3.1.2. Message Properties

Message properties include various attributes and metadata accompanying the message.

Some basic property lists:

Routing key: Used by the exchange to determine how to route the message to the queue (depending on the exchange type).

Message ID: A unique identifier that identifies the message.

Timestamp: Time when the message was created.

Headers: Custom headers containing additional information about the message.

Delivery Mode: Specifies the delivery mode, whether messages should be stored to disk to maintain durability or not.

Expiration: Time when the message is removed if not used.

Priority: Message priority level.

Content type: Describes the format of the message content (e.g. application/json).

Content encoding: Specifies the encoding used for the message content (e.g. utf-8).

3.2. Producer

Producer is an application that sends messages to RabbitMQ. This is the starting point in RabbitMQ's workflow.

Producers typically open one or more TCP connections to exchanges as the application initializes. These connections usually persist as long as their connections or running applications (long-lived connection).

3.3. Bindings

Bindings are relationships established to determine the relationship between exchanges and queues. They specify routing rules to send messages from the exchange to the appropriate queue. A queue can be bound to multiple exchanges and an exchange can have multiple bindings to different queues.

3.4. Exchanges

In RabbitMQ, the exchange acts as a message router. When a producer sends a message to RabbitMQ, the message will be sent to the exchange. The exchange is then responsible for routing the message to the appropriate queue.

RabbitMQ has five different exchange types used for routing messages. Each exchange type will have different parameter usage and binding configurations. Clients can create their own exchanges or use default exchanges provided by the server.

The five basic Exchange types in RabbitMQ are:

  1. Direct Exchange: Routes messages to queues based on the routing key. If the routing key of the message matches the binding key of the queue, the message will be routed to that queue.
  1. Fanout Exchange: Routes all messages to all queues it knows. Ignores routing key.
  1. Topic Exchange: Routes messages to queues based on pattern matching between the routing key and binding key.
  1. Headers Exchange: Routes messages based on message headers instead of routing key. When binding, you can define conditions for headers.
  1. Default Exchange: A special exchange type, every queue that is created is automatically bound to the default exchange with the binding key being the queue name.

In addition, there is one more special exchange type:

Dead Letter Exchange (DLX): This is a special exchange type used to handle messages that cannot be processed by RabbitMQ. When a message cannot be delivered to any queue or is rejected by a consumer, the message will be sent to the DLX. DLX supports error handling and ensures messages are not lost during processing.

3.5. Queues

Queue is a list of messages stored until they are consumed by consumers.

Messages are routed to queues through bindings from exchanges. RabbitMQ allows creating multiple queues with different configurations. Each queue has a name, and optional other properties such as durability, message expiration time and maximum length of messages.

3.6. Consumers

Consumer is an application that receives and processes messages from RabbitMQ. It listens for messages from one or more queues and then processes them.

Consumers can disconnect from RabbitMQ. When RabbitMQ detects a lost connection, message delivery will pause.

4. Applications of RabbitMQ

RabbitMQ has many applications in the following situations:

  1. Build message systems between services in a distributed architecture.
  1. Handle asynchronous background tasks to increase processing speed of user requests.
  1. Ensure reliable transmission of information, ensure no data loss during transmission.
  1. Create queueing systems to evenly distribute workloads to processing agents.
  1. In building push notification systems.

II. Installation on Docker

Guide to installing RabbitMQ on Docker for Magento 2

Steps to install RabbitMQ on Docker for Magento 2:

  1. Create a rabbitmq service and update required variables (env/rabbitmq.env) in docker-compose.yml
...
rabbitmq:
		image: markoshust/magento-rabbitmq:3.9-0
		ports:
		    - "5672:5672"
				- "15672:15672"
		volumes:
		    - rabbitmqdata:/var/lib/rabbitmq
		env_file: env/rabbitmq.env
		environment:
		    - TZ=Asia/Ho_Chi_Minh
...
RABBITMQ_HOST=rabbitmq
RABBITMQ_PORT=5672
RABBITMQ_MANAGEMENT_PORT=15672
RABBITMQ_DEFAULT_USER=magento
RABBITMQ_DEFAULT_PASS=magento
RABBITMQ_DEFAULT_VHOST=/
RABBITMQ_HEALTHCHECK_TIMEOUT=10
  1. Start docker
docker-compose up -d

Here, 15672:15672 allows you to access the RabbitMQ management interface via the browser on port 15672, and 5672:5672 is the port that Magento 2 will connect to send and receive messages. Now you can access the RabbitMQ Management Interface via http://localhost:15672 or http://domain:15672 . Use the username and password same as in rabbitmq.env file (default is guest).
